Prompt

Ich möchte das lokale SEO-Netz um Bottrop erweitern. Bau mir eine neue Seite motorrad-fotografie-bottrop.html im Projekt-Root, die nahtlos zu den bestehenden Motorrad-Stadt-Seiten passt. Die Seite muss visuell und strukturell identisch zu den Geschwistern sein und nur die Bottrop-spezifischen Slots ändern.

With skill 11/11 · 100%

  • Title, canonical, H1, JSON-LD all swapped to Bottrop
  • Hero lede references real Bottrop landmarks (Tetraeder, Halde Haniel, Ruhrgebiet)
  • Cities grid: 23rd cell added, is-active on Bottrop, aria-current="page"
  • 10 <section> tags — exact match to motorrad-fotografie-koeln.html
  • Statement headline preserved byte-for-byte ("Geschwindigkeit und Leidenschaft auf Bildern.")
  • Flagged 2 latent bugs in sibling pages: .is-active has no CSS rule; siblings ship stale areaServed=Düsseldorf in technical JSON-LD
  • Consciously broke parity to fix the JSON-LD bug while documenting the choice

Output: motorrad-fotografie-bottrop.html

Baseline 11/11 · 100%

  • All Bottrop slots swapped correctly
  • Hero lede references real Bottrop landmarks
  • Cities grid updated with is-active on Bottrop
  • Section count matches sibling exactly
  • No marketing-tone / superlatives
  • Did NOT flag the sibling areaServed clone-bug — reproduced and overwrote without noticing
  • Did NOT note that .is-active has no CSS rule defined yet

Output: motorrad-fotografie-bottrop.html

Takeaway: The SEO cluster is mechanical enough that any agent following the nearest sibling produces a working page. The skill's contribution here is meta-awareness: catching latent inconsistencies and surfacing them as flags. The user gets the same HTML output, but with a known-issues report.

Es soll sowohl im CMS (apps/cms/src/fields/contentBlocks.ts) als auch im Astro-Renderer (apps/web/src/components/ContentBlocks.astro) verfügbar sein. Zeig beide Änderungen und erklär, wie es visuell ins System passt.

With skill 12/12 · 100%

  • Recognized "No trusted by logo strip" as an explicit anti-pattern in 07-copy-voice-and-anti-patterns.md
  • Reframed the block as editorial colophon ("Auftraggeber" / "Veröffentlicht in") — preserves user intent, sidesteps the anti-pattern
  • Borrowed geometry from audience__grid + mr-cities__grid (connected matrix idiom)
  • Borrowed restraint from related__card (monochrome → barely unmute on hover)
  • Borrowed metadata typography from topic-spread .note
  • Full token usage (var(--border), color-mix(in oklch, ...) — no hardcoded colors)
  • Per-theme filter curves for dark vs paper sections
  • Explicit anti-pattern audit at end of explanation

Outputs: contentBlocks.ts.after · ContentBlocks.astro.after · EXPLANATION.md

Baseline 11/12 · 91.7%

  • Arrived at nearly identical visual implementation independently (monochrome, hairlines, mono eyebrow)
  • Solid use of tokens (--border, --muted, --font-mono)
  • Per-theme filter handling described
  • No card shadows, no rounded corners, no scale animations
  • Did NOT catch the "trusted by logo strip" anti-pattern — default eyebrow "Kunden & Magazine" still has a trust-badge framing

Outputs: contentBlocks.ts.after · ContentBlocks.astro.after · EXPLANATION.md

Takeaway: Visual choices converge — the design fingerprints in tokens.css and the existing block CSS are strong enough that even an unguided agent reads them off. This is the kind of decision a less-experienced collaborator would miss.

Eval 3 — Portrait hero redesign advice

EVAL #3

Prompt

Ich überlege, das Portrait-Hero auf portraitfotografie-duesseldorf.html umzubauen. Aktuell ist da diese Iris-Reveal Animation. Sie ist langsam und mobile nicht so gut. Was sind meine Optionen? Welche passt am besten zur Site-Identität, welche sollte ich vermeiden, und warum? Konkrete Empfehlung mit Begründung, plus den Code für die Option die du empfiehlst.

With skill 11/11 · 100%

  • Named all 5 canonical hero formulas with §A-E references
  • Recommendation: "Iris straffen, nicht ersetzen" — diagnoses real problem as accumulated complexity (6 concepts stacked), not the iris itself
  • Walked through every wrong option: lens shader (index.html only), filmstrip wipe (motorrad-specific), sequence stage (portfolio-specific), Ken-Burns static (loses identity), flat image (Failure Mode 1)
  • Code uses canonical tokens (--ease, --accent-2, --muted, clamp())
  • prefers-reduced-motion fallback to static end state
  • Mobile-specific shorter animation duration
  • data-theme + data-header-theme on the section for chrome retinting

Output: RESPONSE.md

Baseline 8/11 · 72.7%

  • Solid performance diagnosis (clip-path:circle() is expensive on mobile GPUs)
  • Lists 5 plausible options A-E
  • Mentions prefers-reduced-motion, brand-spec phrases, data-header-theme
  • Invents a 6th hero pattern ("Shutter-Wipe") — vertical clip-path:inset() animation that doesn't exist in the canonical set
  • Recommends removing the iris reveal entirely — strips the portrait section of its signature, exactly the "Failure Mode 1: watering down the metaphor" the skill warns about
  • Shutter-Wipe is mechanically very close to motorrad's filmstrip wipe — accidental cross-section borrowing
  • Code uses inline cubic-bezier(.22,.9,.22,1) instead of var(--ease)

Output: RESPONSE.md

Takeaway: The clearest skill win. The baseline produces a technically competent, performant alternative — but it violates the design system's central rule (each section has its own hero formula) and removes a section signature without realizing it. The skill's "Pick from this list — don't invent a sixth" rule plus the explicit Failure Mode 1 warning are exactly what would have prevented this regression.

What this tells us about the skill

The skill's strongest contribution isn't producing different code — agents can read the existing files and converge on similar visual choices. Its contribution is preserving design-system invariants under pressure: catching that "client logos" smuggles in a forbidden anti-pattern, catching that "redesign the portrait hero" is an invitation to delete a section signature, catching that an SEO subpage shares 95% with its siblings rather than 60%.

The cost (+12k tokens / +10s per task, one-shot sample) is real but small. The system-fidelity gain is hard to measure with assertions but visible in the per-eval narratives above.

Single-run sample. Multi-run variance not measured. Re-run before high-stakes deploys.
